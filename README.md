مرحله ۱: بررسی، تست و مستندسازی APIها با Swagger	
هدف:
آشنایی عملی با سرویس‌های سمت سرور پروژه و بررسی نحوه عملکرد APIها با استفاده از ابزار Swagger.
شرح فعالیت:
در گام اول، به‌منظور بررسی سرویس‌های سمت سرور، وارد محیط Swagger پروژه شدم. Swagger یک ابزار تحت وب برای نمایش، تست و مستندسازی APIها است که امکان بررسی دقیق درخواست‌ها و پاسخ‌ها را فراهم می‌کند.
آدرس Swagger پروژه به صورت زیر در مرورگر وارد شد:
https://edu-api.havirkesht.ir/docs
پس از ورود به این صفحه، لیست کاملی از APIهای پروژه نمایش داده شد که هر یک شامل اطلاعاتی مانند نام سرویس، نوع متد GET)، POST، PUT،  (DELETEو مسیر API بودند.
در ادامه، هر API به‌صورت جداگانه بررسی شد و با استفاده از گزینه Try it out، درخواست‌ها به سرور ارسال گردید. برای هر API، پارامترهای ورودی (در صورت وجود) وارد شده و پاسخ دریافتی از سرور مورد بررسی قرار گرفت. این پاسخ‌ها شامل داده‌های JSON بودند که ساختار خروجی API را مشخص می‌کردند.
در طول این مرحله، APIها از نظر کاربرد در صفحات مختلف سیستم تحلیل شدند و مشخص گردید که هر صفحه سمت کاربر از کدام API برای دریافت یا ارسال داده استفاده می‌کند. همچنین وضعیت پاسخ‌ها (موفق یا ناموفق بودن درخواست‌ها) بررسی شد.

در قالب یک جدول مستندسازی شده و برای استفاده در مراحل بعدی پروژه آماده گردید.
نتیجه مرحله:
در این مرحله تمامی APIهای پروژه با موفقیت بررسی و تست شدند و مستندات لازم برای استفاده از آن‌ها در پیاده‌سازی سمت کاربر تهیه گردید. این مستندات پایه و مبنای مراحل بعدی پروژه، به‌ویژه طراحی و پیاده‌سازی رابط کاربری، قرار گرفت.

در این مرحله با مراجعه به مستندات Swagger، APIهای پروژه شناسایی و بررسی شدند.
هر API از نظر آدرس (Endpoint)، نوع درخواست (Method) و کاربرد مورد تحلیل قرار گرفت و اطلاعات آن در قالب جدول مستندسازی شد.

جمع‌بندی مرحله:
در این مرحله تمامی APIهای مورد نیاز پروژه شناسایی، بررسی و مستندسازی شدند. این مستندات مبنای پیاده‌سازی منطق برنامه و ارتباط با سرور در مراحل بعدی پروژه قرار گرفت.
 

مرحله ۳: پیاده‌سازی صفحات سمت کاربر و اتصال به APIها
هدف:
طراحی و پیاده‌سازی صفحات سمت کاربر و برقراری ارتباط آن‌ها با سرویس‌های سمت سرور.
شرح فعالیت:
در این مرحله پیاده‌سازی بخش فرانت‌اند پروژه انجام شد. برای ساختاردهی صفحات وب از HTML، برای طراحی و استایل‌دهی از CSS و Tailwind CSS و برای پیاده‌سازی منطق و پویایی صفحات از JavaScript استفاده گردید.
به‌منظور برقراری ارتباط با سرویس‌های سمت سرور، کتابخانه Axios برای ارسال و دریافت درخواست‌های HTTP به کار گرفته شد. این ابزار امکان مدیریت ساده و مؤثر درخواست‌ها و پاسخ‌های دریافتی از APIها را فراهم نمود.
آدرس APIهای مورد نیاز هر بخش از سامانه از سامانه آموزشی ارائه‌شده توسط استاد استخراج شد. سپس برای هر صفحه، API مربوطه شناسایی و به‌صورت جداگانه به آن متصل گردید. داده‌های دریافتی از سرور پس از پردازش، در صفحات وب نمایش داده شدند.
در این مرحله تلاش شد ساختار کدها به‌صورت منظم و قابل نگهداری پیاده‌سازی شود و ارتباط بین صفحات و APIها به شکل صحیح برقرار گردد. همچنین صحت دریافت داده‌ها و نمایش اطلاعات در رابط کاربری مورد بررسی قرار گرفت.
نتیجه مرحله:
صفحات سمت کاربر با استفاده از HTML، CSS، JavaScript و Tailwind CSS با موفقیت پیاده‌سازی شدند و ارتباط آن‌ها با سرویس‌های سمت سرور از طریق Axios برقرار گردید. این مرحله پایه اصلی عملکرد پروژه را تشکیل می‌دهد و امکان تعامل کاربر با داده‌های سامانه را فراهم ساخت.
 

مرحله ۴: مدیریت و نمایش خطاها با استفاده از SweetAlert
هدف:
مدیریت خطاهای احتمالی و اطلاع‌رسانی مناسب به کاربر با هدف افزایش تجربه کاربری و شفاف‌سازی وضعیت سیستم.
شرح فعالیت:
در این مرحله، به‌منظور مدیریت خطاها و نمایش پیام‌های مناسب به کاربر، از کتابخانه SweetAlert استفاده شد. این کتابخانه امکان نمایش پیام‌های هشدار، خطا و موفقیت را به‌صورت گرافیکی و کاربرپسند فراهم می‌کند.
در هنگام ارسال درخواست‌ها به APIها از طریق Axios، پاسخ‌های دریافتی از سرور بررسی شدند. در صورتی که درخواست با خطا مواجه می‌شد، نوع خطا تشخیص داده شده و پیام مناسب با استفاده از SweetAlert به کاربر نمایش داده می‌شد.
خطاهای مدیریت‌شده در این مرحله شامل موارد زیر بودند:
خطاهای اعتبارسنجی ورودی کاربر (مانند خالی بودن فیلدها یا ورود اطلاعات نادرست)
خطاهای احراز هویت (مانند نام کاربری یا رمز عبور اشتباه)
خطاهای سمت سرور (مانند خطاهای ۴۰۰ و ۵۰۰)
خطاهای ارتباطی و عدم دسترسی به سرور
برای هر یک از این خطاها، پیام مشخص و قابل فهمی به کاربر نمایش داده شد تا علت بروز مشکل به‌وضوح مشخص شود و کاربر بتواند نسبت به اصلاح ورودی یا تلاش مجدد اقدام نماید.
همچنین در صورت انجام موفق عملیات (مانند ارسال موفق اطلاعات یا دریافت صحیح داده‌ها)، پیام موفقیت مناسب با استفاده از SweetAlert به کاربر نمایش داده شد.
نتیجه مرحله:
با پیاده‌سازی سیستم مدیریت خطا و استفاده از SweetAlert برای نمایش پیام‌ها، تعامل کاربر با سامانه بهبود یافت و خطاها به‌صورت شفاف و قابل درک نمایش داده شدند. این امر موجب افزایش کیفیت رابط کاربری و حرفه‌ای‌تر شدن پروژه گردید.
 

مرحله ۵: ثبت دامنه و پیکربندی DNS
هدف:
ایجاد دامنه اختصاصی برای پروژه و اتصال آن به سرویس ابری به‌منظور آماده‌سازی زیرساخت انتشار پروژه.
شرح فعالیت:
در این مرحله به‌منظور راه‌اندازی دامنه اختصاصی پروژه، ابتدا از طریق سامانه ایرنیک اقدام به ثبت دامنه انجام شد. پس از ثبت موفق دامنه، برای مدیریت DNS و اتصال دامنه به سرویس میزبانی، در سرویس آروان کلود ثبت‌نام انجام شد.
پس از ایجاد حساب کاربری در آروان کلود، دامنه ثبت‌شده به پنل کاربری اضافه گردید و تنظیمات DNS دامنه از حالت پیش‌فرض خارج شد. در ادامه، Name Serverهای ارائه‌شده توسط آروان کلود در پنل ایرنیک ثبت و جایگزین Name Serverهای قبلی دامنه شدند.
پس از اعمال تغییرات، فرآیند انتشار (Propagation) دامنه آغاز شد که ممکن است چند ساعت تا حداکثر ۲۴ ساعت زمان ببرد. در این مدت، دامنه به‌تدریج به سرویس آروان کلود متصل شده و امکان مدیریت کامل DNS فراهم گردید.
در نهایت، پس از تأیید اتصال دامنه، تنظیمات لازم برای مدیریت رکوردهای DNS از طریق پنل آروان کلود انجام شد و دامنه برای استفاده در مراحل بعدی پروژه آماده گردید.
نتیجه مرحله:
دامنه پروژه با موفقیت ثبت شده و از طریق سرویس آروان کلود به‌درستی پیکربندی گردید. این زیرساخت امکان اتصال پروژه به دامنه اختصاصی و ادامه مراحل استقرار را فراهم نمود.
 
 
مرحله ۶: استقرار پروژه بر روی پلتفرم لیارا
هدف:
استقرار پروژه وب بر روی یک پلتفرم ابری به‌منظور اجرای آنلاین و دسترسی عمومی از طریق دامنه اینترنتی.
شرح فعالیت:
در این مرحله به‌منظور استقرار پروژه، از پلتفرم ابری لیارا استفاده شد. ابتدا در وب‌سایت لیارا ثبت‌نام انجام شد و پس از ورود به پنل کاربری، یک برنامه جدید متناسب با نوع پروژه ایجاد گردید.
پس از ایجاد برنامه، سورس‌کد پروژه آماده‌سازی شد و از طریق ابزارهای ارائه‌شده توسط لیارا، فرآیند استقرار (Deploy) پروژه انجام گرفت. در این مرحله، فایل‌ها و تنظیمات مورد نیاز پروژه بر روی سرورهای لیارا بارگذاری شدند و سرویس مربوطه به‌صورت خودکار اجرا گردید.
پس از اتمام فرآیند استقرار، پروژه با موفقیت بر روی زیرساخت لیارا اجرا شد و یک آدرس اینترنتی پیش‌فرض از سوی لیارا برای دسترسی به پروژه در اختیار قرار گرفت. در ادامه، تنظیمات لازم جهت اتصال دامنه ثبت‌شده در مراحل قبل به پروژه مستقرشده انجام شد.
در نهایت، با بررسی عملکرد پروژه از طریق مرورگر، صحت اجرای برنامه و دسترسی آنلاین به آن تأیید گردید.
نتیجه مرحله:
پروژه با موفقیت بر روی پلتفرم لیارا مستقر شد و امکان دسترسی به آن از طریق آدرس اینترنتی فراهم گردید. این مرحله بستر لازم برای انتشار نهایی پروژه و ادامه مراحل توسعه را مهیا کرد.

 

مرحله ۷: اتصال پروژه به GitHub و ارسال کدها (Push)
در این مرحله، هدف انتقال پروژه از محیط محلی (سیستم شخصی) به مخزن GitHub و مدیریت نسخه‌های پروژه با استفاده از سیستم کنترل نسخه Git می‌باشد.
توضیح کلی مرحله
GitHub یک پلتفرم مدیریت مخازن کد است که امکان ذخیره، کنترل نسخه، پیگیری تغییرات و همکاری تیمی را فراهم می‌کند.
در این مرحله، پروژه آماده‌سازی شده و به مخزن GitHub متصل شد تا کدها به‌صورت امن و قابل مدیریت ذخیره شوند.
مراحل انجام کار
1. نصب و تنظیم اولیه Git
ابتدا ابزار Git روی سیستم نصب و از صحت نصب آن اطمینان حاصل شد.
سپس نام کاربری و ایمیل برای ثبت صحیح commitها تنظیم گردید.
git --version
git config --global user.name 
git config --global user.email 
2. آماده‌سازی پروژه
پروژه در محیط VS Code آماده شد و فایل .gitignore ایجاد گردید تا پوشه‌ها و فایل‌های غیرضروری مانند node_modules و dist در مخزن GitHub آپلود نشوند.
3. ایجاد مخزن Git و ثبت نسخه اولیه
در پوشه اصلی پروژه، مخزن Git ساخته شد و فایل‌ها به‌عنوان نسخه اولیه ثبت شدند.
git init
git add .
git commit -m "Initial commit"
git branch -M main
4. ایجاد Repository در GitHub
در وب‌سایت GitHub یک مخزن جدید با نام پروژه ایجاد شد.
5. اتصال پروژه محلی به GitHub (Remote)
پس از ساخت مخزن در GitHub، پروژه محلی به مخزن آنلاین متصل شد.
git remote add origin 
در صورتی که آدرس remote اشتباه باشد، از دستور زیر برای اصلاح آن استفاده شد:
git remote set-url origin 
6. ارسال پروژه به GitHub (Push)
در این مرحله، پروژه برای اولین بار به مخزن GitHub ارسال شد.
git push --set-upstream origin main
7. رفع خطای احتمالی Fetch First
در صورتی که مخزن GitHub دارای commit قبلی باشد و اجازه push داده نشود، ابتدا تغییرات دریافت شده و سپس ارسال انجام شد.
git pull --rebase origin main
git push --set-upstream origin main
8. ارسال تغییرات بعدی
برای هر تغییر جدید در پروژه، مراحل زیر انجام شد:
git add .
git commit -m "message"
git push
نتیجه نهایی
در پایان این مرحله:
پروژه با موفقیت به مخزن GitHub متصل شد.
تمامی فایل‌های پروژه در GitHub ذخیره گردید.
امکان مدیریت نسخه‌ها و پیگیری تغییرات فراهم شد.
پروژه آماده استفاده در مراحل بعدی مانند CI/CD شد.
جمع‌بندی
مرحله هفتم نقش مهمی در مدیریت پروژه داشت و با اتصال پروژه به GitHub، بستر لازم برای توسعه، نگهداری و استقرار خودکار پروژه فراهم گردید.

 
 



مرحله ۸: فعال‌سازی گواهی SSL و اتصال دامنه با استفاده از آروان‌کلود و لیارا
هدف:
افزایش امنیت وب‌سایت از طریق فعال‌سازی گواهی SSL و برقراری ارتباط امن (HTTPS) برای دامنه پروژه.
شرح فعالیت:
در این مرحله به‌منظور تأمین امنیت ارتباط بین کاربر و سرور، از سرویس گواهی SSL آروان‌کلود استفاده شد. ابتدا در پنل آروان‌کلود برای دامنه ثبت‌شده، درخواست صدور گواهی SSL ثبت گردید. پس از انجام تنظیمات لازم و تأیید دامنه، فرآیند فعال‌سازی گواهی آغاز شد.
در ادامه، برای اتصال دامنه به پروژه مستقرشده در لیارا، دامنه مورد نظر در پنل لیارا اضافه گردید. پس از افزودن دامنه، رکوردهای DNS ارائه‌شده توسط لیارا دریافت شد و این رکوردها در بخش مدیریت DNS آروان‌کلود ثبت شدند.
پس از اعمال تنظیمات DNS و گذشت زمان لازم برای انتشار تغییرات (Propagation)، ارتباط دامنه با سرویس لیارا برقرار شد. در نهایت با فعال شدن گواهی SSL، وب‌سایت از طریق پروتکل HTTPS در دسترس قرار گرفت و امنیت تبادل اطلاعات تضمین شد.
نتیجه مرحله:
گواهی SSL با موفقیت فعال شد و دامنه پروژه به‌صورت امن به سرویس لیارا متصل گردید. در نتیجه، وب‌سایت با استفاده از پروتکل HTTPS قابل دسترسی بوده و سطح امنیت آن افزایش یافت.

 
 


مرحله ۹: پیاده‌سازی فرایند CI/CD 
در این مرحله، به‌منظور خودکارسازی فرآیند استقرار پروژه، قابلیت CI/CD پیاده‌سازی شد تا پس از هر بار اعمال تغییرات در کد پروژه، نسخه جدید به‌صورت خودکار بر روی سرور اعمال شود.
توضیح کلی مرحله
CI/CD یک روش استاندارد برای یکپارچه‌سازی و تحویل مداوم نرم‌افزار است که باعث کاهش خطای انسانی، افزایش سرعت توسعه و به‌روزرسانی سریع پروژه می‌شود.
در این پروژه، با استفاده از امکانات مخزن گیت‌هاب، فرآیند استقرار خودکار پیاده‌سازی گردید.
مراحل انجام کار
ابتدا یک فایل پیکربندی مخصوص اجرای CI/CD در مسیر زیر از مخزن پروژه ایجاد شد:
.github/workflows/liara.yaml
در این فایل، مراحل اجرای خودکار پروژه تعریف شد به‌گونه‌ای که با هر بار ارسال تغییرات (Push) به مخزن گیت‌هاب، فرآیند استقرار به‌صورت خودکار اجرا شود.
نام برنامه در فایل پیکربندی تنظیم شد تا فرآیند استقرار به‌درستی بر روی برنامه مورد نظر انجام شود.
برای افزایش امنیت، کلید دسترسی API مربوط به سرویس استقرار به‌صورت مستقیم در کد قرار داده نشد و از قابلیت Secrets در تنظیمات مخزن گیت‌هاب استفاده شد.
مقدار کلید API با نام مشخص در بخش Secrets تعریف شد تا در زمان اجرای خودکار، بدون افشای اطلاعات حساس مورد استفاده قرار گیرد.
پس از ذخیره فایل پیکربندی و ارسال آن به مخزن، فرآیند CI/CD به‌صورت خودکار فعال شد.
با هر بار ایجاد تغییر در پروژه و ارسال آن به مخزن گیت‌هاب، عملیات استقرار به‌صورت خودکار اجرا شده و نسخه جدید پروژه بر روی سرور اعمال گردید.
نتیجه نهایی
با پیاده‌سازی این مرحله:
فرآیند استقرار پروژه کاملاً خودکار شد.
نیازی به اعمال دستی تغییرات روی سرور وجود ندارد.
هر تغییر جدید بلافاصله پس از ارسال به مخزن، روی سرور اعمال می‌شود.
امکان مشاهده تاریخچه اجرای مراحل استقرار از طریق بخش Actions در گیت‌هاب فراهم شد.
جمع‌بندی
در این مرحله، با استفاده از قابلیت CI/CD، فرآیند توسعه و استقرار پروژه به‌صورت خودکار و استاندارد پیاده‌سازی شد که باعث افزایش سرعت، دقت و پایداری پروژه گردید.
